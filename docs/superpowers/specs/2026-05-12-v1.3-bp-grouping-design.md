# DL-Processor v1.2 — Business Process Grouping for Review Pending

**Status:** approved 2026-05-12 (brainstormed with Ali through real-world v1.1 usage)
**Branch:** TBD — `feat/v1.2-bp-grouping` off master (after v1.1.0 ship)
**Estimated effort:** ~2 weeks of subagent-driven execution

## Goal

Replace the per-field-per-unit row view on Review Pending with **Business Process (BP) cards** that group related field changes from the same compare run, label them semantically, and surface SF workflow state inline so users review with full context.

## Background

After v1.1 shipped, real-world use surfaced a recurring problem:

When a single business event occurs in SF (e.g., a resale to a new buyer), it produces three field changes on the same unit at the **same timestamp**:

- `buyer_name` changes (new buyer)
- `purchase_price_aed` changes (new price)
- `procedure_number` changes (new DLD application)

The current Review Pending page shows these as **3 separate rows**. Reviewing them one at a time fragments context, and the reviewer has to mentally re-group them.

Moreover, the existing view doesn't surface SF's workflow state. **SF is the process log** — `Status`, `Current Step Name`, `Current Step Assigned Name`, `Pre-registration status`, `DLD Process Status`, `Comments` all describe whether a BP is:

- **complete** (safe to review and approve into `master_data`)
- **in progress** (a workflow is running; verify in SF first)
- **has a DLD issue** (customer info dispute with DLD; do not approve until resolved)
- **rejected** (BP was killed in SF; pending changes are informational)

Reviewing without this context risks approving changes SF is still processing — or worse, approving against rejected BPs that should never have been written to `master_data`.

v1.2 reshapes Review Pending around BPs and brings SF context inline.

## Decisions (from brainstorming)

| # | Decision |
|---|---|
| 1 | Group `pending_change` rows by shared `source_snapshot_id` (the compare run that produced them). Exact, defensible grouping. |
| 2 | Label each group by its field-set pattern (Resale, Buyer correction, Price amendment, Status update, Procedure update, Area correction, Multi-field update). Pure function on the set of changed fields. |
| 3 | Collapsible-card UI — header always visible, expand for per-field details. Card-level **Approve all** / **Reject all** plus per-row Approve/Override/Reject. |
| 4 | SF state classifier: **READY** (`Status=Completed` AND `dld_process_status≠'Submitted to DLD'`) / **IN_PROGRESS** (`Status=In Progress`) / **DLD_ISSUE** (`dld_process_status='Submitted to DLD'`) / **REJECTED** (`Status=Rejected`) / **NO_SF_ROW** (no matching sf_booking row). |
| 5 | When state ≠ READY, the card-level **Approve all** is disabled with an explanatory banner; per-row Approve/Override remain available so a careful reviewer can still pass individual fields after manual SF check. REJECTED state has **Acknowledge** only (no approve). |
| 6 | Show full SF context on every card: BP name, status, current step, assigned to, pre-reg status, DLD process status, BP created date, payment date, comments, procedure number. Read live from `sf_booking` at render time. |
| 7 | Filter bar above the cards: Project, Tower, BP type, SF state, Assigned to, Procedure #, Date range, free-text search. AND-composed. Default filter = SF state ≠ REJECTED + Date range = Last 30 days. |
| 8 | Add `[Open in SF →]` deep link per card using `sf_booking.booking_record_id`. |
| 9 | Add 3 new columns to `sf_booking`: `current_step_name`, `current_step_assigned_name`, `comments`. Migration 006. Update `src/salesforce.js` importer to extract them by header name (case-insensitive). |

## Data model

### Schema migration 006 — new columns on `sf_booking`

```sql
ALTER TABLE sf_booking ADD COLUMN current_step_name           TEXT;
ALTER TABLE sf_booking ADD COLUMN current_step_assigned_name  TEXT;
ALTER TABLE sf_booking ADD COLUMN comments                    TEXT;
```

No CHECK constraints — values are open-ended SF strings. Also update `db/schema.sql` so fresh DBs get the columns directly.

### Schema migration 007 — widen `audit_log.action` CHECK

To support BP-level audit entries, widen the action CHECK constraint:

```sql
-- Rebuild-table pattern (SQLite can't ALTER CHECK in place)
-- Old action set: ('approve', 'override', 'reject', 'auto_apply', 'learn_alias')
-- New action set: add 'approve_bp', 'reject_bp', 'acknowledge_bp'
```

### No new tables

BP grouping is computed at query time from existing `pending_change.source_snapshot_id`. No `bp_group` table needed. Synthetic `bp_id = source_snapshot_id || '_' || project_id || '_' || unit_number_norm` is used in IPC payloads.

### Importer update

`src/salesforce.js` (or wherever the SF .xlsx parser is) must extract three new columns. Likely Excel header names:

- `Current Step Name` (or `Current Step: Name`)
- `Current Step: Assigned Name` (or `Current Step Assigned Name`)
- `Comments`

Add a case-insensitive header-tolerance pass: normalize each Excel header to lowercase + strip spaces/colons, then match against the column map. Existing extracted columns (applicant_name, status, etc.) all continue to work.

## BP classifier (pure function)

```js
const AUDIT_FIELDS_SET = new Set(['buyer_name', 'purchase_price_aed', 'status', 'procedure_number', 'area_sqm']);

function classifyBp(fieldSet) {
  const has = (...fs) => fs.every(f => fieldSet.has(f));
  // Resale = buyer + price + procedure (optionally + status)
  if (has('buyer_name', 'purchase_price_aed', 'procedure_number')) return 'Resale';
  if (fieldSet.size === 1 && has('buyer_name'))         return 'Buyer correction';
  if (fieldSet.size === 1 && has('purchase_price_aed')) return 'Price amendment';
  if (fieldSet.size === 1 && has('status'))             return 'Status update';
  if (fieldSet.size === 1 && has('procedure_number'))   return 'Procedure update';
  if (fieldSet.size === 1 && has('area_sqm'))           return 'Area correction';
  return 'Multi-field update';
}
```

First-match-wins. Field-set is a `Set` for O(1) lookup. Adding new BP categories later is a single new case before `Multi-field update`.

## SF state classifier (pure function)

```js
function classifyState(sfRow) {
  if (!sfRow) return 'NO_SF_ROW';
  if (sfRow.status === 'Rejected') return 'REJECTED';
  if (sfRow.dld_process_status === 'Submitted to DLD') return 'DLD_ISSUE';
  if (sfRow.status === 'Completed') return 'READY';
  return 'IN_PROGRESS';
}
```

Fallback `IN_PROGRESS` for unknown values is **conservative** — safer to warn than to silently approve. New SF status values won't accidentally become "ready" until explicitly classified.

### State badges (color)

| State | Badge | Banner | Approve all |
|---|---|---|---|
| READY | green | (none) | enabled |
| IN_PROGRESS | yellow | "⚠ BP is still running in SF. Verify completion before approving." | disabled |
| DLD_ISSUE | orange | "⚠ DLD has issue with customer info. Resolve in SF first." | disabled |
| REJECTED | red | "✗ BP was rejected in SF. Changes are informational only." | replaced with **Acknowledge** |
| NO_SF_ROW | gray | "⚠ No SF record found for this unit. Cannot verify BP state." | enabled but double-confirm prompt |

### Expand behavior

When expanded, the card shows:
- Each field change as an editable row with `New (edit)` input (override) + per-row Approve / Reject buttons (same as v1.1 Needs review tab)
- Full SF context strip stays visible
- Comments expand to full text (was truncated to 2 lines in collapsed view)

### Card actions

- **Approve all** — atomic transaction: approve every field row (using each row's current `New (edit)` value, override or default), write per-field audit_log entries, write umbrella `action='approve_bp'` entry.
- **Reject all** — atomic transaction: reject every field row, write per-field audit_log entries with `action='reject'`, write umbrella `action='reject_bp'` entry.
- **Acknowledge** (REJECTED only) — write umbrella `action='acknowledge_bp'` audit_log entry, mark field rows `decision='rejected'` so they fall out of the queue. No `master_data` writes.
- **Open in SF →** — opens `https://<sobha-sf-domain>/<booking_record_id>` in browser via Electron's `shell.openExternal`. Domain configurable in Settings.
- **Expand / Collapse** — toggles card body visibility.

## Filter bar

```
[Project ▼]  [Tower ▼]  [BP type ▼]  [SF state ▼]  [Assigned to ▼]
[Procedure # ___]  [Date range ▼]  [Search ___]
```

| Filter | Behavior |
|---|---|
| Project | Multi-select. Populated from `dld_project` + SF-only projects (same UNION as v1.1 top-bar). |
| Tower | Multi-select. Populated from distinct `sf_booking.tower_name` for the selected projects. |
| BP type | Multi-select. Values: the 7 labels from the classifier. |
| SF state | Multi-select. Values: READY / IN_PROGRESS / DLD_ISSUE / REJECTED / NO_SF_ROW. |
| Assigned to | Multi-select. Distinct `current_step_assigned_name` values. |
| Procedure # | Free text, exact match on the unit's current `sf_booking.procedure_number`. |
| Date range | Today / Last 7 / Last 30 (default) / Last 90 / All time / Custom. Filters by `pending_change.decided_at` (or `proposed_at` for pending rows). |
| Search | Free text. Matches against unit number, buyer name (old or new), comments. |

All filters AND-compose. Reset button clears all. Default filters on load: **SF state ≠ REJECTED**, **Date range = Last 30 days**.

## Audit log

Group-aware. When card-level Approve all fires, the transaction inserts:

1. **N per-field audit_log entries** (one per field): `action='approve'` (or `'override'` if `New (edit)` was modified), `change_id=<pending_change.change_id>`, `source='review_pending'`.
2. **One umbrella audit_log entry**: `action='approve_bp'`, `change_id=NULL`, `unit_number_norm=<unit>`, `project_id=<pid>`, `new_value='<source_snapshot_id>'`, `user_note='Resale / 3 fields'` (or whatever the BP label is).

Symmetric handling for Reject all (`action='reject_bp'`) and Acknowledge (`action='acknowledge_bp'`).

The umbrella entry's role: queryable history of "how many BPs were approved this month" by `SELECT COUNT(*) FROM audit_log WHERE action='approve_bp' AND ts >= ...`. The per-field entries remain queryable for field-level history (per-unit side panel still works as-is).

Migration 007 widens the `action` CHECK to include the three new values.

## IPC additions

- `dlp:review:list-bps(opts)` — returns an array of BP groups, each:
  ```js
  {
    bp_id: 'src_snap_42_proj_3_unit_A1207',    // synthetic
    project_id, project_name, tower_name, unit_number_norm,
    label: 'Resale',                            // from classifyBp
    state: 'READY',                             // from classifyState
    sfContext: {
      bp_name, status, current_step_name, current_step_assigned_name,
      pre_reg_status, dld_process_status, bp_created_date, payment_date,
      procedure_number, comments, booking_record_id
    },
    rows: [{ change_id, field_name, old_value, proposed_value, override_value, decision, ... }, ...]
  }
  ```
  Replaces flat-row rendering. Backend joins `pending_change` with `sf_booking` (latest snapshot) and groups in SQL.
- `dlp:review:approve-bp(bpId, overrides)` — atomic Approve all. `overrides` = `{ [change_id]: 'user_typed_value', ... }` for fields the user overrode.
- `dlp:review:reject-bp(bpId)` — atomic Reject all.
- `dlp:review:acknowledge-bp(bpId)` — REJECTED state only.

Existing per-row `dlp:review:approve` / `dlp:review:reject` / `dlp:review:teach-alias` handlers remain — they back the in-card per-row buttons.

## Out of scope (v1.3+ follow-ups)

1. **Auto-classification by SF status taxonomy** — semantic labels per Pre-reg status / Current step values (e.g., "Awaiting customer documents"). Needs more real-data observation.
2. **Multi-step workflow state machine** — knowing the order DLD Registration Initiation → Validate Request → Customer Documents → DLD Completion with Payment, and surfacing "stuck at step X for N days".
3. **"Assigned to me" pseudo-filter** — needs the app to know the staff's SF user identity.
4. **DLD drift detection** — still deferred from v1.1 (needs compare.js extractor refactor).
5. **Revert from History page** — one-click undo of an approval.
6. **Value-based heuristics** — Levenshtein-distance buyer-name similarity for distinguishing "minor edit" vs "different person".
7. **SF deep-link domain configurability** — Settings field for the Salesforce instance URL pattern. For v1.2 we hard-code the standard Sobha SF instance.
