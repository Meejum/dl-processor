# DL-Processor v1.2 — Patch-Based Updates

**Status:** approved 2026-05-12 (re-scoped v1.2 after BP-grouping deferred to v1.3)
**Branch:** `feat/v1.2-patch-system` (off master, branched from `9aa5deb`)
**Estimated effort:** ~1 week (10 tasks, single developer)

## Goal

Ship app updates as small zip patches that Ali shares with the team via email / OneDrive / network share. Users apply the zip from inside the app (native UI) — no full 78 MB reinstall every release.

## Background

v1.1 ships as a 78 MB Windows installer. Every update requires Ali to:
1. Build a new `DL-Processor Setup 1.X.Y.exe`
2. Upload to Cloudflare Pages OR share via OneDrive
3. Each team member: uninstalls v1.X-1, runs new installer, says "No" to data removal, waits

This is expensive (every cycle is 78 MB × N team members) and high-friction (full reinstall feels weighty for small changes). For a team that runs a monthly reconciliation cycle, the friction discourages incremental improvements between cycles.

v1.2 introduces a patch-update mechanism:

- Build script produces `dlp-patch-vA-to-vB.zip` (~5-15 MB instead of 78 MB)
- App has an in-app **Apply patch** action (native UI, no iframe/HTML)
- User picks zip → app verifies → swaps the `app.asar` archive at next launch
- Schema migrations run automatically post-restart (existing v1.1 migration framework)

## Decisions

| # | Decision |
|---|---|
| 1 | **Patch = `app.asar` replacement.** Native node_modules stay constant (better-sqlite3 binary unchanged across v1.x with Electron 28). Patch zip contains the new `app.asar` (~30 MB raw → ~5-15 MB zipped) plus a manifest. |
| 2 | **Manifest** has: `from_version_min`, `to_version`, `built_at`, `asar_sha256`, `asar_size`, `app_id` (sanity check). Patch only applies if installed `from_version >= from_version_min`. |
| 3 | **Apply UI is native renderer DOM** — overlay modal (same pattern as `unit-history-panel.js`), NOT an iframe with srcdoc. No HTML files. |
| 4 | **File swap happens at next launch via a bundled helper batch script** (`patch-apply.cmd` bundled in `resources\`). The app stages the new asar to `%LOCALAPPDATA%\Programs\DL-Processor\resources\app.asar.pending`, writes a flag, quits, and launches `patch-apply.cmd` which swaps the asar and relaunches `DL-Processor.exe`. |
| 5 | **Rollback** is automatic on swap failure (the `.bak` of the old asar is kept until next successful launch). Manual rollback via a "Revert last patch" action in Settings. |
| 6 | **No code signing.** The patch zip is unsigned; users verify it came from Ali by his sharing method (email, OneDrive). v1.2 adds a SHA-256 hash display so users CAN verify if they want. |
| 7 | **First patch-capable version is v1.2.0 itself.** v1.1 users do one last full reinstall to v1.2.0. From v1.2.0 onwards, all updates (including the v1.3 BP grouping work) are patches. |

## Patch zip format

```
dlp-patch-v1.2.0-to-v1.3.0.zip
├── manifest.json
└── app.asar               (the new Electron app archive)
```

`manifest.json`:
```json
{
  "app_id": "ae.sobha.dl-processor",
  "from_version_min": "1.2.0",
  "to_version": "1.3.0",
  "built_at": "2026-05-12T10:00:00Z",
  "asar_sha256": "<hex digest>",
  "asar_size": 31250812,
  "release_notes": "BP grouping for Review Pending; SF state badges."
}
```

The `app_id` check prevents accidentally applying a patch for the wrong app. `from_version_min` lets patches span multiple versions (e.g., v1.2.0→v1.4.0 if no breaking change).

## Build script — `scripts/build-patch.js`

Invoked: `node scripts/build-patch.js --from 1.2.0 --to 1.3.0 [--notes "..."]`

Steps:
1. Run `npm run dist` (produces full installer + the unpacked `dist-electron/win-unpacked/resources/app.asar`).
2. Read `dist-electron/win-unpacked/resources/app.asar`.
3. Compute SHA-256 + size.
4. Build manifest with current package.json version as `to_version`.
5. Zip manifest + asar into `dist-electron/dlp-patch-v{from}-to-v{to}.zip`.
6. Print the SHA-256 + size + zip path to console.

No git history walking, no diffing — the patch is always a full `app.asar` replacement. Smaller than the .exe because the patch doesn't include Electron's runtime, native modules, NSIS installer overhead, or Windows resources.

## Apply UI — `electron/renderer/patch-modal.js`

Native overlay modal at `document.body` level (not iframe). Pattern: same as `unit-history-panel.js` from v1.1.

Sidebar entry:
```
HELP
  ⬆ Apply update (patch)
```

Flow:
1. User clicks **⬆ Apply update**.
2. Modal opens. Heading: "Apply DL-Processor update". One button: **Choose patch file…**.
3. User picks `.zip` via native file picker.
4. App calls `dlp:patch:probe-zip(zipPath)`. Backend:
   - Extracts manifest.json into memory
   - Verifies `app_id`, `from_version_min ≤ current version`, `asar_sha256` matches contained `app.asar`
   - Returns `{ ok: true, manifest, currentVersion }` or `{ ok: false, error }`.
5. Modal updates with verification result:

```
┌─ Apply DL-Processor update ───────────────────────────────┐
│ Patch verified.                                            │
│                                                            │
│ Current version:  1.2.0                                    │
│ Update to:        1.3.0                                    │
│ Built:            2026-05-12 14:30                         │
│ Size:             12.4 MB (compressed)                     │
│ SHA-256:          a3f2b1...e7d9 (verify with Ali if you    │
│                   want before applying)                    │
│                                                            │
│ Release notes:                                             │
│   BP grouping for Review Pending; SF state badges.         │
│                                                            │
│ ⚠ The app will close and restart automatically.            │
│   Your data (Desktop\DL-Processor\) is NOT affected.       │
│                                                            │
│              [Cancel]              [Apply & Restart]       │
└────────────────────────────────────────────────────────────┘
```

6. User clicks **Apply & Restart**. App calls `dlp:patch:apply(zipPath)`. Backend:
   - Extracts `app.asar` from zip to `<install-dir>\resources\app.asar.pending`
   - Writes a marker file `<install-dir>\resources\.patch-pending`
   - Launches `<install-dir>\resources\patch-apply.cmd` with arguments (PID of current app, paths)
   - Calls `app.quit()`
7. `patch-apply.cmd`:
   - Waits up to 10 seconds for the main app process to exit (polls via `tasklist`)
   - Renames `app.asar` → `app.asar.bak`
   - Renames `app.asar.pending` → `app.asar`
   - Deletes `.patch-pending` marker
   - Launches `DL-Processor.exe`
   - Exits
8. On relaunch: app reads new code from the swapped asar. v1.1 migration framework runs any new migrations. Done.

If `patch-apply.cmd` fails midway:
- If `app.asar` already renamed to `.bak` but the new asar rename failed → cmd restores `.bak` to `.asar`
- If app re-launches with the OLD asar still in place (because cmd never ran) → user just retries

## IPC handlers

- `dlp:patch:probe-zip(zipPath)` — Read manifest, verify SHA-256 of contained asar, return verification result. Read-only; doesn't touch install dir.
- `dlp:patch:apply(zipPath)` — Stage the new asar, write marker, launch helper script, schedule quit. Returns `{ scheduled: true }` immediately.
- `dlp:patch:revert-last()` — If `<install-dir>\resources\app.asar.bak` exists, restore it via same helper script. Returns `{ scheduled: true }`.

Exposed in `electron/preload.js` as:
```js
patch: {
  probeZip:    (args) => ipcRenderer.invoke('dlp:patch:probe-zip', args),
  apply:       (args) => ipcRenderer.invoke('dlp:patch:apply',     args),
  revertLast:  ()     => ipcRenderer.invoke('dlp:patch:revert-last')
}
```

## Testing strategy

Test target: 322 → ~340 (+18).

**Unit tests** (Node-side, isolated from Electron):
- Manifest parsing (5 cases — valid, missing fields, wrong app_id, wrong from_version_min, malformed JSON)
- SHA-256 verification (3 cases — match, mismatch, missing asar in zip)
- Version compare for `from_version_min` (3 cases — exact match, newer than min, older than min)
- Zip extraction (2 cases — single zip, malformed zip)
- Stage to `.pending` filename (manual mock filesystem; 2 cases)
- Revert from `.bak` (2 cases — bak exists, bak missing)

**Manual smoke** (renderer not unit-testable):
- Build a v1.2.0 patch (`node scripts/build-patch.js --from 1.2.0 --to 1.2.0-test`)
- Install v1.2.0 fresh on test machine
- Open app → Apply update → pick the zip
- Verification shows correctly
- Apply → app quits + restarts with new asar
- Confirm new version in title bar
- Test revert: Apply a junk patch → expect verification fail; or apply real patch, then "Revert last" → app restarts with old asar

## Security considerations

The patch is unsigned. Risks:
- A malicious party could craft a patch zip with malware in the asar
- Mitigation: app verifies SHA-256 against manifest's stated hash; user gets a final SHA-256 to compare with what Ali sent

For v1.2 we rely on the trust chain: Ali → team via SOBHA's email/OneDrive. v1.4 (audit hardening) might add Authenticode signing.

## Out of scope (defer)

1. **Delta patches** — diffing two asars for partial replacement. Asar isn't designed for diffing; full replacement is simpler and the size is acceptable.
2. **Native module updates** — better-sqlite3 binary stays the same across v1.x. If we change Electron version (e.g., v3.0), we'll need a separate full installer release.
3. **Multi-step patch chains** — v1.2.0 → v1.5.0 in one go. v1.2 just supports single-step `from_min → to`; users running multiple versions behind apply patches in sequence.
4. **Background/auto-detection of patches** — v1.2 is manual file-picker only. Auto-download from Cloudflare Pages is a v1.3+ enhancement.
5. **macOS/Linux patches** — Windows-only.

## Release

- **v1.2.0** ships as a full `.exe` installer (no patches yet because v1.1 doesn't know how to apply them).
