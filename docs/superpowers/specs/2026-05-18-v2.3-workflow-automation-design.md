# v2.3 — Workflow Automation: Design

**Author:** Ali Alghumlasi (with Claude)
**Date:** 2026-05-18
**Status:** Approved — ready for implementation planning
**Prior art:** v2.1 audit hardening (tier-2, audit chain), v2.2 native dashboard (renderer-DOM pattern), `src/auto-approve.js` (alias-based auto-apply)

## 1. Goal

Reduce manual review time on the monthly DLD ↔ Salesforce reconciliation by:

1. Letting users automate repetitive decisions through a declarative rules engine (no scripting).
2. Surfacing genuinely unusual changes as visible anomaly badges, distinct from the v2.1 tier-2 workflow block.
3. Removing per-row friction with queue-level and BP-card-level bulk actions.
4. Making month-over-month activity changes legible from the dashboard.
5. Letting users preview what a compare will produce before committing to a review.

Success is measured by: rule firings replacing manual approvals (target ≥30% of pending changes auto-resolved within 4 weeks of v2.3 use), zero rule-driven reverts in the first month, and team feedback that bulk approve / dry-run / trending tile reduce cycle time.

## 3. Architecture

### 3.1 Rules engine is the spine

The rules engine is the **single entry point for automated decisions** during compare and drift detection. Three currently-scattered decision points get unified:

- `src/auto-approve.js` (alias-based `shouldAutoApprove`) → becomes built-in rule `R-1000`.
- v2.1 `src/tier2.js` (`isTier2`) → stays a pure predicate but is exposed as a *condition* in the rules engine; tier-2 rows still hit the existing justification modal.
- New built-in anomaly rules (price / area / novel-procedure / late-buyer-change) → ship as default-enabled rules.

User-authored rules and built-in rules live in the same `automation_rule` table and the same evaluation pipeline. Built-ins are visible but locked (cannot be deleted; can be disabled by the user, which is logged to `audit_log`).

**Naming convention.** Each rule has an integer `id` (primary key). The display name is `R-<id>` (e.g., `R-1000`). The `audit_log.source` value is `rule:<id>` (e.g., `rule:1000`) for firings and `rule:<id>-disable` for disable events. Throughout this spec, `R-NNN` refers to the display form and `rule:NNN` refers to the audit source form.

### 3.2 Pipeline order

In `src/compare-drift.js` (and the SF branch of compare), after a candidate `pending_change` row is built but before it is persisted:

```
candidate change + context
        │
        ▼
ruleEngine.evaluate(change, context, rules)
   │
   │  Two-pass evaluation per candidate change:
   │    Pass 1 — accumulate anomalies: every enabled rule whose action
   │             is 'flag_anomaly' and whose WHEN matches contributes a
   │             reason to change.anomaly. Anomaly rules NEVER short-
   │             circuit. Final severity = max of contributed severities.
   │    Pass 2 — terminal action: scan enabled rules in priority order
   │             for the first match whose action is NOT 'flag_anomaly'
   │             (i.e., one of auto_approve / auto_reject /
   │             auto_acknowledge_bp / skip). That action wins.
   ▼
{ action, anomaly?, note?, rule_id?, reasons[] }
   │
   ├─ action = 'auto_apply'         → write pending_change(status='auto_applied', anomaly=…) + audit_log(source='rule:<id>')
   ├─ action = 'auto_reject'        → write pending_change(status='auto_rejected', anomaly=…) + audit_log
   ├─ action = 'auto_acknowledge_bp' → only valid for BP-grouped rows; otherwise treated as no-match
   ├─ action = 'skip'               → do not persist (intentional drop)
   └─ no terminal match             → write pending_change(status='pending', anomaly=…) — normal review flow, 🚨 badge if anomaly populated
```

Note that `anomaly` data can co-exist with any terminal action — a row may be auto-applied AND carry an anomaly badge, in which case the badge surfaces only on History (since the row is no longer in Review Pending). The crypto chain from v2.1 covers every rule-driven audit_log row identically to manual approvals — revert remains one-click.

## 4. Feature B — Custom rules engine

### 4.1 Data model

New table:

```sql
CREATE TABLE automation_rule (
  id              INTEGER PRIMARY KEY,
  name            TEXT NOT NULL,
  enabled         INTEGER NOT NULL DEFAULT 1,
  priority        INTEGER NOT NULL,        -- ASC, first match wins
  when_json       TEXT NOT NULL,           -- predicate tree
  then_json       TEXT NOT NULL,           -- action descriptor
  builtin         INTEGER NOT NULL DEFAULT 0, -- 1 = locked-built-in
  created_at      TEXT NOT NULL,
  created_by      TEXT,                    -- audit_log.user equivalent
  applied_count   INTEGER NOT NULL DEFAULT 0,
  revert_count    INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_automation_rule_priority ON automation_rule(enabled, priority);
```

Priority bands: user rules `1-999`, built-ins `1000+`. Lower number = higher priority. First matching rule wins (short-circuit evaluation).

### 4.2 WHEN predicate shape

```json
{
  "op": "and",
  "clauses": [
    {"field": "change_type", "operator": "=",  "value": "BUYER_MISMATCH"},
    {"field": "alias_exists", "operator": "=", "value": true},
    {"op": "or", "clauses": [
      {"field": "delta_pct",   "operator": "<=", "value": 5},
      {"field": "delta_abs",   "operator": "<=", "value": 1000}
    ]}
  ]
}
```

- Logical ops: `and`, `or`. Nesting allowed (validated at save time, max depth 3).
- Leaf operators: `=`, `!=`, `>`, `<`, `>=`, `<=`, `in`, `regex`.
- `regex` operator value must be a string compiled at load time; invalid regex → rule disabled with error in `audit_log`.

### 4.3 Field allowlist

Fields accessible inside WHEN clauses (fixed list; anything else is a validation error):

```
change_type, field, delta_pct, delta_abs, alias_exists,
bp_type, sf_state, project_id, project_name,
tier2, source, unit_number_norm, procedure_number
```

Resolved by the engine from the candidate `change` object and a context object that wraps `db`, the current snapshot ids, and any cached aliases. The engine itself does not touch the DB during evaluation — context is built once per compare run.

### 4.4 THEN action shape

```json
{
  "action": "auto_approve",
  "note": "alias match, sub-threshold",
  "anomaly_severity": null
}
```

Allowed actions: `auto_approve`, `auto_reject`, `flag_anomaly`, `auto_acknowledge_bp`, `skip`.

- `flag_anomaly` requires `anomaly_severity ∈ {warn, high}`.
- `auto_acknowledge_bp` is only valid when the change is part of a BP card; non-BP rows fall through to the next rule.
- `note` is optional, max 200 chars, written to `audit_log.user_note`.

### 4.5 Built-in seeded rules

Migration 009 seeds these four rules with `builtin=1`:

| id  | name | priority | WHEN | THEN |
|-----|------|----------|------|------|
| 1000 | Auto-approve buyer-alias matches | 1000 | `change_type=BUYER_MISMATCH AND alias_exists=true AND tier2=false` | `auto_approve` |
| 1001 | Flag large price increases | 1001 | `field=purchase_price AND delta_pct>25` | `flag_anomaly, severity=high` |
| 1002 | Flag large area changes | 1002 | `field=area_sqm AND delta_pct>10` | `flag_anomaly, severity=high` |
| 1003 | Auto-acknowledge cancelled BPs | 1003 | `bp_type=REGISTRATION AND sf_state=cancelled` | `auto_acknowledge_bp` |

Users can disable built-ins (writes an `audit_log` row with `source='rule:builtin-disable'`) but not delete them.

### 4.6 UI (renderer)

New sidebar entry "Automation" → native renderer-DOM page (v2.2 pattern).

Layout (see brainstorm mockup `rules-editor.html`):

- Toolbar: `+ New rule`, `Import` (disabled in v2.3 — placeholder), `Export` (disabled).
- Rule list: one row per rule with checkbox (enable/disable), name, applied/revert counts.
- Expanded row shows WHEN / THEN in monospace with syntax-coloring (purple `WHEN`/`AND`/`OR`, red `THEN`, teal operators, code-style values).
- `+ New rule` opens an inline editor: clause builder with field/operator/value dropdowns, AND/OR chips between clauses, THEN action dropdown. Save validates against the field allowlist and predicate schema.
- Each rule row links to a per-rule History view (filtered audit_log where `source='rule:R-NNN'`).

### 4.7 CLI

```
node . rules list                      # show all rules with status + counts
node . rules enable <id>
node . rules disable <id>
node . rules test <id> --snapshot=<n>  # dry-evaluate rule against snapshot n
```

## 5. Feature C — Smart anomaly flags

### 5.1 Data

`pending_change` gains one new column:

```sql
ALTER TABLE pending_change ADD COLUMN anomaly TEXT;  -- JSON or NULL
```

Shape when non-null:

```json
{
  "severity": "high",
  "reasons": [
    {"rule_id": 1001, "code": "price_jump", "detail": "delta_pct=42.3%"}
  ]
}
```

Multiple anomaly reasons can accumulate on one row. Per the two-pass evaluation in §3.2, anomaly rules (`flag_anomaly`) never short-circuit and all matching ones contribute reasons in priority order. The terminal action is then chosen independently in pass 2. Severity rolls up to the max of contributed reasons (`high` > `warn`).

### 5.2 Built-in anomaly predicates (`src/anomaly.js`)

Pure functions, callable independently of the rule engine (also surfaced through the seeded rules above so users can disable them):

1. `priceJump(change) → boolean` — `field=purchase_price && delta_pct > 25`
2. `areaJump(change) → boolean` — `field=area_sqm && delta_pct > 10`
3. `novelProcedure(change, ctx) → boolean` — `field=procedure_number && !seenInTrailingMonths(value, 6)`
4. `lateBuyerChange(change, ctx) → boolean` — `field=buyer_name && bp_state='completed'`

All four are reachable both as rule conditions (when the rules engine fires) and as a fast-path scanner that the renderer can run on already-loaded rows for live filtering.

### 5.3 UI

- Review Pending grid: new "Flags" column showing 🚨 (high) or ⚠️ (warn). Sortable.
- Clicking a flag opens the existing side panel (v2.2) with an "Anomaly reasons" section listing each reason's detail.
- Filter bar gains a 9th filter: `Flagged: any / high / warn / none`.

### 5.4 Distinction from tier-2 (must stay clear)

| | Anomaly flag | Tier-2 gate |
|---|---|---|
| Visibility | Always visible badge in queue | Visible only when user clicks Approve |
| Effect | None — informational | Blocks approval until justification ≥10 chars |
| Co-occurrence | Can co-occur on same row | — |
| Origin | Anomaly rules (built-in + user `flag_anomaly`) | v2.1 settings thresholds (price %, AED, area %) |
| Storage | `pending_change.anomaly` JSON | `audit_log.tier2=1`, `audit_log.user_note` |

The two systems remain independent in code: `anomaly.js` knows nothing about tier-2, and `tier2.js` is unchanged.

## 6. Feature A — Bulk operations

### 6.1 Queue-level bulk approve / reject

Review Pending toolbar gains a row-selection model:

- Filter bar already supports 8 dimensions (v2.0). Toolbar adds:
  - `[Select all matching]` — checks every visible row across pagination.
  - `[Approve N]` / `[Reject N]` — counts shown live.
- Confirmation modal lists the count, a 5-row sample, and any tier-2 rows in the selection.
- If any selected rows are tier-2, a single combined justification modal fires (≥10 chars), reusing v2.1's existing `tier2.js` predicate and modal — same pattern as the BP-card "Approve all" path.
- Backend operation is one transaction per chunk of 50 rows; failure aborts the chunk but commits prior chunks. Result modal shows `N applied, M failed` with a "view failed" link.

### 6.2 BP-card multi-selection

Existing BP cards (v2.0) get a checkbox in the card header. New top-of-page banner appears when ≥1 card is selected: `[Approve 3 cards · Reject 3 cards · Clear]`. Tier-2 aggregation works as above — one combined modal across all selected cards.

### 6.3 Audit trail

Each row in a bulk action still writes its own `audit_log` row (the crypto chain doesn't batch). Bulk operations share a single `batch_id` (UUID v4 generated client-side, written to `audit_log.source` as `bulk:<UUID>`) so a future History UI can show "this was part of a 23-row bulk approve" — schema-light approach, no new column. Using `source` instead of `user_note` avoids the collision with tier-2 justifications, which already own `user_note`. The CHECK widening in §9 covers the `bulk:%` prefix.

When a bulk action contains tier-2 rows, the shared justification text goes to each tier-2 row's `user_note` as today (v2.1 behavior); the `bulk:<UUID>` ties them together via `source`. Non-tier-2 rows in the same batch carry the same `source='bulk:<UUID>'` with `user_note=NULL`.

## 7. Feature D — Cross-month trending tile

### 7.1 Query

```js
function getTrendingProjects(db, { minBaseline = 5, ratioThreshold = 2.0 } = {}) {
  // Returns: [{ project_id, project_name, this_month, trailing_avg, ratio }, ...]
  // Buckets pending_change by month using imported_at of the source dld_snapshot.
  // trailing_avg = mean of previous 6 calendar months (excluding current).
  // Filter: this_month >= minBaseline AND ratio >= ratioThreshold.
  // Sort by ratio DESC.
}
```

Settings expose `minBaseline` (default 5) and `ratioThreshold` (default 2.0).

### 7.2 UI

Dashboard adds one tile in the top row, between the existing totals tile and project cards:

```
┌─ ⚠ Unusual activity (3 projects) ────────────────┐
│ Project ONE     23 pending  (avg 7 / 6mo, 3.3×) │
│ Project SIX     18 pending  (avg 5 / 6mo, 3.6×) │
│ Project 19      11 pending  (avg 4 / 6mo, 2.8×) │
└──────────────────────────────────────────────────┘
```

Empty state ("No unusual activity") hides the tile entirely. Each row links to Project Compare filtered to the current month's snapshot.

## 8. Feature E — Pre-compare report (dry-run)

### 8.1 CLI

```
node . compare --dry-run [--source=dld|sf|both]
```

Runs the full compare pipeline including rule evaluation, but inside a SAVEPOINT that is rolled back. Outputs to stdout (and to `output/dry-run-<timestamp>.txt`):

```
DRY-RUN — would write 142 pending_change rows + 38 audit_log auto_apply rows

By change_type:
  BUYER_MISMATCH    47  (32 auto-approve, 12 pending, 3 flagged-high)
  PRICE_DRIFT       28  (18 pending, 8 flagged-high, 2 tier-2)
  ...

By project:
  ONE      54
  SIX      31
  ...

Top 10 sample rows:
  …
```

### 8.2 Desktop modal

After a snapshot import, before the user clicks "Apply pending" for the first time on that snapshot, a modal shows:

```
Preview — this compare would create:
  142 pending changes
    • 38 auto-approved by rules
    • 11 with 🚨 high anomaly flag
    •  8 needing tier-2 justification
  Top changes: …
  [Continue] [Cancel]
```

Implementation reuses the same dry-run path with `--source=both` and a JSON output mode. The JSON shape is:

```json
{
  "would_write": {"pending_change": 142, "audit_log_auto_apply": 38},
  "by_change_type": [{"type": "BUYER_MISMATCH", "total": 47, "auto_approve": 32, "pending": 12, "anomaly_high": 3, "anomaly_warn": 0, "tier2": 0}, ...],
  "by_project":     [{"project_id": 1, "name": "ONE", "total": 54}, ...],
  "samples":        [{"unit": "...", "field": "...", "old": "...", "new": "...", "decision": {"action": "auto_apply", "rule_id": 1000}}, ...]
}
```

The CLI's text mode is rendered from the same JSON; the renderer modal reads the JSON directly via IPC `dlp:compare:dry-run`.

## 9. Schema migration

Single migration file `src/migrations/2026-05-18-009-automation.js`, idempotent in one pass:

1. `CREATE TABLE IF NOT EXISTS automation_rule (…)` with the index.
2. `ALTER TABLE pending_change ADD COLUMN anomaly TEXT` guarded by a column-exists check (same idiom as migration 008).
3. Widen `audit_log.source` CHECK to include `rule:%` and `bulk:%` prefixes. CHECK widening follows the v2.0 migration 005 idiom: rename old table, create new table with widened CHECK, copy rows, drop old, rename — wrapped in `db.transaction`.
4. Insert 4 built-in rules with `INSERT OR IGNORE` on `id` so re-runs are no-ops.

No backfill is needed: `pending_change.anomaly` defaults to NULL for existing rows. Existing rule-engine machinery does not retroactively evaluate old snapshots — only new compares.

## 10. IPC additions

New handlers in `electron/main.js`:

```
dlp:rules:list                  → all rules with applied/revert counts
dlp:rules:get(id)               → full rule row
dlp:rules:create(rule)          → validate + insert; returns new id
dlp:rules:update(id, patch)     → name / when / then / enabled / priority
dlp:rules:delete(id)            → rejects if builtin=1
dlp:rules:test(id, snapshotId)  → dry-evaluate, returns [{change, decision}]
dlp:trending:get()              → trending tile rows
dlp:compare:dry-run(opts)       → JSON summary (re-uses CLI dry-run)
dlp:bulk:approve(rowIds, just?) → batched apply with optional tier-2 justification
dlp:bulk:reject(rowIds)         → batched reject
```

All handlers go through the existing preload bridge pattern; signatures wrap a single options object so additive changes are backwards-compatible.

## 11. Testing

Test count target: 428 → ~480 (52 new). Highlights:

- **Rule engine (25 tests):** every operator (`=`, `!=`, `>`, `<`, `>=`, `<=`, `in`, `regex`); `and`/`or` nesting depth 1, 2, 3; priority ordering; first-match-wins; disabled rules skipped; built-in priority floor enforced; malformed WHEN rejected; unknown field rejected; regex compile failure disables rule with audit_log entry; field allowlist boundary cases.
- **Anomaly (10 tests):** each of the 4 built-in predicates with positive/negative/edge cases; accumulation when multiple anomaly rules match.
- **Trending (6 tests):** baseline filter, ratio filter, sort order, empty-state, settings-override defaults, snapshot timestamp bucketing edge cases (month boundary).

- **Dry-run (6 tests):** SAVEPOINT rollback verified, summary counts match real compare counts, source=dld vs sf vs both, JSON vs text output, no audit_log rows written.

Integration test: `test/v2.3-end-to-end.test.js` imports a fixture snapshot, runs compare with seeded user rule, asserts `pending_change` + `audit_log` shapes including the `rule:<id>` source format and the two-pass anomaly+terminal split.

## 13. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| User writes a rule that auto-approves everything | First-match-wins + built-in low-priority safety rule; per-rule applied_count + revert_count visible; built-in revert (v2.1) still works |
| Rule evaluation slows compare significantly | Engine is pure-function and runs over already-loaded candidate changes — measured separately in tests with a 1000-row fixture; budget <100ms per 1000 rows |
| Built-in rule disabled silently | Disabling any rule writes `audit_log` with `source='rule:R-NNN-disable'`; History page surfaces this |
| Anomaly + tier-2 conceptual confusion | Distinct columns, distinct UI affordances, documented in Settings help text |
| Bulk approve hides individual mistakes | Each row writes its own audit_log; v2.1 one-click Revert works per-row; bulk batch_id lets future Undo-batch be added without schema change |

## 16. Open questions resolved during brainstorming

| Question | Decision |
|----------|----------|
| Rules engine: DSL or JSON tree? | **JSON tree** (security, secure validation, easy UI) |
