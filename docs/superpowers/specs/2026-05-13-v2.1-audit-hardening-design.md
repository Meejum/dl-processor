# DL-Processor v2.1 — Audit & Compliance Hardening

**Status:** approved 2026-05-13 (brainstormed with Ali post v2.0 ship)
**Branch:** `feat/v2.1-audit-hardening` off master after v2.0.0
**Estimated effort:** ~1 week subagent-driven execution
**Test target:** 390 → ~420 (+30 new)

## Goal

Five audit-hardening features that prepare the audit_log for compliance reviews and give reviewers an escape hatch when an approval was wrong:

1. **One-click Revert** on global History page rows — undoes an approval, restores prior `master_data` value
2. **"Approved by" user attribution** — every new `audit_log` entry stamped with the human who made the change
3. **Cryptographic chain on audit_log** — tamper-evident: each row's hash includes the prior row's hash; modifying past rows breaks the chain
4. **Tier-2 approval gates** — large changes (price >threshold, area >threshold) require justification text before commit
5. **Excel audit export** — compliance-grade `.xlsx` export from the History page

Distributed via the v1.2 patch system (~3-5 MB patch zip from v2.0.0 to v2.1.0).

## Background

v2.0 delivered the BP grouping + inline review UX. Real-world use will surface two needs:

1. **Wrong approvals need an undo.** Today's audit_log records every change, but there's no one-click revert. Users have to manually `UPDATE master_data` + write a corresponding audit row by hand. Slow and error-prone.
2. **Compliance teams will eventually audit the auditor.** They need:
   - WHO approved each change (currently NULL — no user attribution)
   - PROOF the log wasn't modified after the fact (tamper evidence)
   - FLAGGED high-magnitude changes that should have gotten extra scrutiny
   - EXPORTABLE evidence in a format their tools accept (Excel, not SQLite)

v2.1 addresses both — single-week milestone, no new tables (just columns), no breaking changes.

## Decisions (from brainstorming)

| # | Decision |
|---|---|
| 1 | Revert = **one-step back** (Cmd-Z model). Restores `master_data` to the audit row's `old_value`. Only `approve` / `override` / `approve_bp` / prior `revert` entries are revertable (`auto_apply` / `reject` / `learn_alias` / `acknowledge_bp` don't modify `master_data` so revert is meaningless for them). |
| 2 | User attribution = **Settings field if set, else OS user** (`os.userInfo().username`). New `audit_log.user` column. Forward-only — existing rows from v1.1/v2.0 stay NULL. |
| 3 | Crypto chain = **backfill + forward**. Migration 008 walks existing audit_log in `(ts, audit_id)` order, computes hashes, writes them. v2.1 onward chains forward at every `writeAuditLog` call. SHA-256. Genesis `prev_hash` = 64 zeros. |
| 4 | Tier-2 gates = **strong warning + required justification** (no password block). Settings holds thresholds (price %, price absolute, area %). Modal prompts justification text → stored in `user_note`. `audit_log.tier2` boolean flag. |
| 5 | Excel export = **button on History page** next to existing CSV export. Honors current filters. xlsx output. |

## Schema — Migration 008

Single migration handling four column adds + CHECK widen + chain backfill. Idempotent.

```sql
-- 008a — column adds (safe to re-run, guarded by PRAGMA table_info check)
ALTER TABLE audit_log ADD COLUMN user      TEXT;
ALTER TABLE audit_log ADD COLUMN tier2     INTEGER NOT NULL DEFAULT 0;
ALTER TABLE audit_log ADD COLUMN prev_hash TEXT;
ALTER TABLE audit_log ADD COLUMN row_hash  TEXT;

-- 008b — widen action CHECK to include 'revert' (rebuild-table pattern, same as
-- migrations 003, 005, 007). Adds 'revert' alongside the existing
-- approve/override/reject/auto_apply/learn_alias/approve_bp/reject_bp/acknowledge_bp.

-- 008c — backfill hashes for existing rows
-- Walk audit_log in (ts, audit_id) order. For each row:
--   prev_hash = previous row's row_hash (or 64-zero genesis for row 1)
--   row_hash  = sha256(prev_hash || canonicalize(row_content))
-- canonicalize = stable JSON serialization of all audit_log columns
-- except prev_hash + row_hash themselves.
```

Hash input format (to be implemented in `src/audit-hash.js`):

```js
function canonicalize(row) {
  return JSON.stringify({
    audit_id:         row.audit_id,
    ts:               row.ts,
    project_id:       row.project_id,
    unit_number_norm: row.unit_number_norm,
    table_name:       row.table_name,
    field:            row.field,
    old_value:        row.old_value,
    new_value:        row.new_value,
    action:           row.action,
    source:           row.source,
    change_id:        row.change_id,
    user_note:        row.user_note,
    user:             row.user,
    tier2:            row.tier2
  });
}

function computeRowHash(prevHash, row) {
  const input = (prevHash || '0'.repeat(64)) + canonicalize(row);
  return crypto.createHash('sha256').update(input).digest('hex');
}
```

The hash explicitly INCLUDES the `audit_id` so re-ordering rows breaks the chain.

## Feature A — Revert

### IPC

```js
// electron/main.js
ipcMain.handle('dlp:audit:revert', (e, { auditId }) => withDb(db => revertAuditEntry(db, auditId)));
```

### Backend — `src/commands/audit-revert.js`

```js
const REVERTABLE_ACTIONS = new Set(['approve', 'override', 'approve_bp', 'revert']);

function revertAuditEntry(db, auditId) {
  const row = db.prepare('SELECT * FROM audit_log WHERE audit_id = ?').get(auditId);
  if (!row) throw new Error('audit row not found: ' + auditId);
  if (!REVERTABLE_ACTIONS.has(row.action)) {
    throw new Error('cannot revert action=' + row.action +
      ' — only approve/override/approve_bp/revert can be reverted');
  }
  if (row.table_name !== 'master_data') {
    throw new Error('cannot revert non-master_data row (table_name=' + row.table_name + ')');
  }

  const tx = db.transaction(() => {
    // Restore master_data.field to row.old_value
    const prov = MASTER_DATA_PROVENANCE[row.field];
    if (!prov) throw new Error('unknown field for revert: ' + row.field);
    db.prepare(`
      UPDATE master_data
      SET ${row.field} = ?, ${prov.source} = 'staff', ${prov.decidedAt} = datetime('now')
      WHERE project_id = ? AND unit_number_norm = ?
    `).run(row.old_value, row.project_id, row.unit_number_norm);

    // Write new audit_log entry
    writeAuditLog(db, {
      projectId:       row.project_id,
      unitNumberNorm:  row.unit_number_norm,
      tableName:       'master_data',
      field:           row.field,
      oldValue:        row.new_value,    // we're undoing this
      newValue:        row.old_value,    // restored to this
      action:          'revert',
      source:          'review_pending',
      changeId:        null,
      userNote:        'revert of audit_id=' + auditId,
      user:            currentUser(db)
    });
  });
  tx();
}
```

## Feature C — Cryptographic chain

### Migration 008 backfill

```js
function backfillHashes(db) {
  const rows = db.prepare(`
    SELECT * FROM audit_log
    WHERE row_hash IS NULL
    ORDER BY ts, audit_id
  `).all();
  let prevHash = null;
  // If there are already chained rows, pick up from the last one
  const lastChained = db.prepare(`
    SELECT row_hash FROM audit_log
    WHERE row_hash IS NOT NULL
    ORDER BY ts DESC, audit_id DESC LIMIT 1
  `).get();
  if (lastChained) prevHash = lastChained.row_hash;
  const upd = db.prepare('UPDATE audit_log SET prev_hash = ?, row_hash = ? WHERE audit_id = ?');
  const tx = db.transaction(() => {
    for (const row of rows) {
      const ph = prevHash || '0'.repeat(64);
      const rh = computeRowHash(ph, row);
      upd.run(ph, rh, row.audit_id);
      prevHash = rh;
    }
  });
  tx();
}
```

### writeAuditLog change

After INSERT, immediately compute + UPDATE `prev_hash` + `row_hash`. The hash is computed on the just-inserted row (including its auto-assigned `audit_id`). All inside the same `db.transaction()` so partial chains can't exist.

### Verify-on-demand (out of scope per Q3 = b, NOT c)

A "Verify audit log integrity" Settings button is NOT in v2.1. The chain exists and is computed correctly; verification is a v2.2+ follow-up (1-2 hours of work — defer until compliance asks).

## Feature D — Tier-2 gates

### Settings

```
Audit & approval thresholds
  Tier-2 price threshold (%):       [10  ]

  Tier-2 area threshold (%):        [5   ]
  Your name (for audit attribution): [_____]
```

App config keys: `tier2_price_pct`, `tier2_price_abs`, `tier2_area_pct`, `audit_user`.

### Threshold check

`src/commands/review-pending.js`'s `approvePending` gains a check before the master_data write:

```js
function isTier2(field, oldValue, newValue, thresholds) {
  if (field === 'purchase_price_aed') {
    const oldN = Number(oldValue);
    const newN = Number(newValue);
    if (!Number.isFinite(oldN) || !Number.isFinite(newN)) return false;
    const absDelta = Math.abs(newN - oldN);
    const pctDelta = oldN === 0 ? 1 : Math.abs((newN - oldN) / oldN) * 100;
    return pctDelta > thresholds.tier2_price_pct || absDelta > thresholds.tier2_price_abs;
  }
  if (field === 'area_sqm') {
    const oldN = Number(oldValue);
    const newN = Number(newValue);
    if (!Number.isFinite(oldN) || !Number.isFinite(newN)) return false;
    const pctDelta = oldN === 0 ? 1 : Math.abs((newN - oldN) / oldN) * 100;
    return pctDelta > thresholds.tier2_area_pct;
  }
  return false;
}
```

### UI — justification modal

When the renderer detects a tier-2 change (computed in the renderer from the row + current thresholds), the approve flow shows a justification modal BEFORE calling `dlp:review:approve`:

```
⚠ Tier-2 change requires justification

  Unit:     SOR-A1207
  Field:    purchase_price_aed
  Change:   1,034,366  →  1,170,827  (+13.2%)
  Reason this large change is correct:
  [_______________________________]
  [_______________________________]
  (required, min 10 characters)

[Cancel] [Approve with justification]
```

The justification text passes through to `audit_log.user_note`. `audit_log.tier2 = 1` for that row.

For card-level Approve all on a BP that contains a tier-2 row, the modal opens for EACH tier-2 row in sequence (or a single combined justification — pick simpler: ONE justification for the BP, applied to all tier-2 rows in it).

### History display

Tier-2 rows show a `⚠` icon + the justification truncated to 60 chars (full text on hover or in the per-unit panel).

## Feature E — Excel audit export

### IPC

```js
ipcMain.handle('dlp:audit:export-xlsx', async (e, opts) => { /* ... */ });
```

### Implementation

Uses the existing `xlsx` library (already in `package.json` dependencies for SF import). Builds a single sheet workbook:

```js
const wb = XLSX.utils.book_new();
const headers = [
  'Timestamp', 'User', 'Project', 'Unit', 'Field',
  'Old value', 'New value', 'Action', 'Source',
  'Tier-2', 'Justification', 'Row hash'
];
const rows = auditQuery.globalHistory(db, opts);   // honors current filters
const data = [headers, ...rows.map(r => [
  r.ts, r.user || '', r.project_name, r.unit_number_norm, r.field,
  r.old_value, r.new_value, r.action, r.source,
  r.tier2 ? '⚠ TIER-2' : '', r.user_note || '', r.row_hash || ''
])];
const ws = XLSX.utils.aoa_to_sheet(data);
XLSX.utils.book_append_sheet(wb, ws, 'Audit Log');
XLSX.writeFile(wb, savePath);
```

### UI

History page filter bar grows from `[Apply] [Export CSV]` to `[Apply] [Export CSV] [Export Excel]`. Same filter-shape contract — exports honor whatever filters are active.
