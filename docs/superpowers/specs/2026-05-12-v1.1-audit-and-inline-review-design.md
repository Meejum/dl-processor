# DL-Processor v1.1 — Audit Log + Inline Review + UI Cleanup

**Status:** approved 2026-05-12 (brainstormed with Ali)
**Branch:** `feat/v1.1-audit-and-inline-review` (off `0243637`, the tip of `feat/electron-desktop`)
**Release tag:** `v1.1.0` (single combined release; v1.0 is never separately published)
**Estimated effort:** ~2 weeks of subagent-driven execution

## Goal

Turn the registration team's monthly reconciliation from a multi-step CSV roundtrip into a single-screen workflow with a complete audit trail. Specifically:

1. Replace the `review-pending` → CSV → `apply-pending` two-step with an inline approve/reject/override page that writes straight to the live DB.
2. Add `audit_log` so every change to live data is queryable per row per field, and surface it via a global History page + per-unit history side panel.
3. Cut BUYER_MISMATCH false-positives by auto-normalizing buyer names at compare time + a learnable `buyer_alias` table.
4. Treat every import as a potential drift event — every silent overwrite leaves an `auto_applied` notification row + an audit log entry, so users can see what changed in the last import without being blocked by it.
5. Fix the "All projects" dropdown bug (currently lists only DLD-imported projects, missing SF-only and `sales_unit`-only projects).
6. Remove the sidebar collapse button (always full-width with icon + text).
7. Add a metadata + impact-summary modal before any DB import.
8. Final sanity-audit pass before release.

## Background

v1.0 (`feat/electron-desktop`, 51 commits, smoke-test green 2026-05-12) packaged the existing CLI as a Windows desktop app with zero-config Desktop install. The reconciliation flow inside the app is still the legacy two-step:

- `compare` writes `pending_change` rows when DLD and SF disagree.
- `review-pending` renders an HTML page; user downloads a decisions CSV and edits it manually.
- `apply-pending` reads the edited CSV and writes the approvals into `sales_unit`.

Real-world friction:

- Two-step roundtrip — users repeatedly forget to run `apply-pending` after editing the CSV.
- BUYER_MISMATCH dominates the queue with transliteration/whitespace/case false positives that recur every month.
- No change-history view — "why did Unit 501's buyer change last month?" requires git-log spelunking on backup zips.
- Drift goes silent — re-importing a DLD report with a corrected price silently overwrites the live value with no record.
- The "All projects" dropdown in the top bar omits Salesforce-only projects, confusing users who filter by project.

v1.1 collapses the two-step into one screen and makes every data mutation permanently queryable.

## Design decisions (from brainstorming)

| # | Decision |
|---|----------|
| 1 | On approve, the `pending_change` row is **kept** (marked `disposition='approved'`) — full review history is queryable forever |
| 2 | Inline override = single editable cell per row (the proposed new value); audit log records `override` action with the user's typed value |
| 3 | Imports always overwrite; drift entries are `auto_applied` pending_change rows + audit log entries (notifications, not blockers) |
| 4 | `audit_log` records every change to live data, per row per field — user actions + auto-applied drift + apply writes |
| 5 | Review Pending page has two tabs: `Needs review` (actionable) and `Drift log` (read-only auto-applied) |
| 6 | Compare auto-normalizes buyer names + new `buyer_alias` learning table; right-click → "Always treat these as equal" |
| 7 | Both a global `📜 History` page (with filters + CSV export) AND a per-unit history side panel reachable from any page |
| 8 | Sidebar collapse button removed entirely; sidebar is always full-width with icon + text |
| 9 | Import DB shows metadata + impact summary before commit (file info, what's inside, current-DB counts, ⚠ confirm) |

## 1. Data model

### New table — `audit_log`

```sql
CREATE TABLE audit_log (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          TEXT    NOT NULL DEFAULT (datetime('now')),
  project_id  INTEGER,
  unit_id     INTEGER,                       -- FK to sales_unit; nullable for non-unit events
  table_name  TEXT    NOT NULL,              -- 'sales_unit' | 'buyer_alias' | ...
  field       TEXT    NOT NULL,              -- 'price' | 'buyer_name_en' | ...
  old_value   TEXT,
  new_value   TEXT,
  action      TEXT    NOT NULL,              -- 'approve' | 'override' | 'reject' | 'auto_apply' | 'learn_alias'
  source      TEXT    NOT NULL,              -- 'review_pending' | 'import_dld' | 'import_sf' | 'apply_pending'
  pending_id  INTEGER,                       -- FK to pending_change row if applicable
  user_note   TEXT
);
CREATE INDEX audit_log_unit_ts    ON audit_log (unit_id,    ts DESC);
CREATE INDEX audit_log_project_ts ON audit_log (project_id, ts DESC);
CREATE INDEX audit_log_pending    ON audit_log (pending_id);
```

Append-only; rows never updated or deleted. Per-row size is small (~200 bytes typical). For a registration team processing 2k units/month with ~50 changes/month, the table grows ~12k rows/year.

### New table — `buyer_alias`

```sql
CREATE TABLE buyer_alias (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id  INTEGER,                       -- NULL = global alias (applies to every project)
  variant     TEXT    NOT NULL,              -- normalize(raw_a), e.g. 'alghumlasi'
  canonical   TEXT    NOT NULL,              -- normalize(raw_b), e.g. 'alghumlasi'
  display     TEXT    NOT NULL,              -- pretty form for the UI, e.g. 'Ali Alghumlasi'
  created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
  created_by  TEXT,
  UNIQUE (project_id, variant)
);
CREATE INDEX buyer_alias_lookup ON buyer_alias (project_id, variant);
```

`variant` and `canonical` are stored already-normalized (lowercased, punctuation stripped, transliteration applied) so the compare-time lookup is a single indexed equality scan.

### Modified table — `pending_change`

Added columns (existing rows back-fill via migration defaults):

| Column | Type | Default | Meaning |
|---|---|---|---|
| `change_type` | TEXT NOT NULL | `'MISMATCH'` | `MISMATCH` \| `DLD_DRIFT` \| `SF_DRIFT` |
| `disposition` | TEXT NOT NULL | `'pending'` | `pending` \| `approved` \| `rejected` \| `auto_applied` |
| `override_value` | TEXT | NULL | What the user typed before approving (NULL if they accepted the proposed value as-is) |
| `reviewed_at` | TEXT | NULL | Set when disposition transitions away from `pending` |
| `reviewer_note` | TEXT | NULL | Optional free-text from the reviewer |

### Schema migration

New module `src/migrations/index.js` runs at every app startup. Tracks applied migrations in a `schema_migration` table (`id TEXT PRIMARY KEY, name TEXT, applied_at TEXT`).

```js
const MIGRATIONS = [
  { id: '2026-05-12-001-audit-log',         sql: '<CREATE TABLE audit_log + indexes>' },
  { id: '2026-05-12-002-buyer-alias',       sql: '<CREATE TABLE buyer_alias + index>' },
  { id: '2026-05-12-003-pending-change-v2', sql: '<ALTER TABLE pending_change ADD COLUMN ...>' },
  { id: '2026-05-12-004-buyer-alias-seed',  sql: '<INSERT INTO buyer_alias (built-in transliteration map)>' }
];
```

- Idempotent — re-running is safe; only un-applied migrations execute.
- Each migration wrapped in a single transaction; failure rolls back cleanly.
- Migrations run BEFORE any IPC handler responds, blocking the renderer with a "Upgrading database…" splash if >2 seconds.

## 2. Compare pipeline changes

### Drift detection

On `import-dld` and `import-sf`, the existing upsert path gains a pre-write diff:

```text
for each incoming row matched to an existing sales_unit:
  for each tracked field in AUDIT_FIELDS:
    if incoming_value != current_value:
      // 1. Overwrite live row (today's behavior — unchanged)
      // 2. Insert pending_change(change_type='DLD_DRIFT' (or 'SF_DRIFT'),
      //    disposition='auto_applied', old_value, new_value)
      // 3. Insert audit_log(action='auto_apply', source='import_dld' (or 'import_sf'),
      //    old_value, new_value, pending_id = <the pending row above>)
```

Tracked fields are a whitelist in new module `src/audit-fields.js`:

```js
const AUDIT_FIELDS = ['price', 'area_sqft', 'buyer_name_en', 'buyer_name_ar', 'sale_date', 'payment_plan'];
```

- New rows (no prior record) → silent insert. No drift entry. No audit entry.
- Deletes (a row disappears from a fresh import) → out of scope for v1.1; handled in a follow-up release.

### Buyer-name normalization (`src/normalize-name.js`)

```js
function normalizeName(raw) {
  if (!raw) return '';
  let s = String(raw).trim().toLowerCase();
  s = s.replace(/^(mr|mrs|ms|miss|dr|eng|sheikh|sh)\s*\.?\s+/i, '');  // strip titles
  s = s.replace(/[.\-_,]/g, ' ');                                      // punctuation → space
  s = s.replace(/\s+/g, ' ').trim();                                   // collapse whitespace
  s = applyTransliterationMap(s);                                      // Arabic-Latin variants
  return s;
}
```

Built-in transliteration map ships with ~50 known Sobha-context pairs extracted from the current month's BUYER_MISMATCH rows (e.g., `al ghumlasi` ↔ `alghumlasi`, `al-` prefix collapse, `gh` ↔ `g` where unambiguous).

The same function is used at compare time AND at `buyer_alias` insert time, so the `variant` column is always already-normalized.

### Buyer-alias lookup (`src/alias-lookup.js`)

At compare time, BEFORE flagging `BUYER_MISMATCH`:

```text
1. dld_norm = normalize(dld_buyer_name)
2. sf_norm  = normalize(sf_buyer_name)
3. if dld_norm == sf_norm: status = MATCH (no pending row)
4. else: look up buyer_alias rows where variant = dld_norm AND (project_id = ? OR project_id IS NULL):
       if any row has canonical = sf_norm: status = MATCH (alias absorbed it)
5. else: status = BUYER_MISMATCH (pending row created)
```

Project-scoped aliases take precedence over global aliases (same `variant` resolves first to the project-specific row if one exists).

### Alias learning

On the Review Pending page, BUYER_MISMATCH rows expose a `🔗 Teach alias` action. Clicking it (in a single transaction):

```sql
INSERT INTO buyer_alias (project_id, variant, canonical, display, created_by)
VALUES (?, normalize(dld_name), normalize(sf_name), sf_name, 'user');

-- Auto-approve this row AND every other visible BUYER_MISMATCH row on the page
-- with the same normalized pair:
UPDATE sales_unit SET buyer_name_en = ? WHERE unit_id IN (...);
UPDATE pending_change SET disposition='approved', reviewed_at=NOW() WHERE id IN (...);

-- Plus one audit_log(action='learn_alias') + one audit_log(action='approve') per auto-approved row.
```

The page refreshes the tab counts and removes the auto-approved rows from view.

## 3. Review Pending UX

### Page structure (two tabs)

The Review Pending page replaces the old HTML-export workflow. Project filter sits above the tabs:

```
┌─ Review Pending ────────────────────────────────────────────────┐
│ Project: [All projects ▼]                                        │
│                                                                   │
│ ┌─────────────────────┬──────────────────┐                       │
│ │ Needs review (12)   │ Drift log (340)  │                       │
│ ├─────────────────────┴──────────────────┴──────────────────────┐│
│ │ ...tab content...                                              ││
│ └────────────────────────────────────────────────────────────────┘│
└───────────────────────────────────────────────────────────────────┘
```

### Tab 1 — `Needs review`

| Column | Behavior |
|---|---|
| `☐` checkbox | Multi-select for batch actions |
| `Unit + Project` | Stacked label; clickable → opens per-unit History side panel |
| `Field + Type` | E.g., `price MISMATCH` / `buyer_en BUYER` |
| `Current` | Read-only — current `sales_unit` value, source labeled (SF) |
| `New (edit)` | Editable input prefilled with the proposed value; debounced 300ms |
| `Actions` | `✓ Approve` / `✗ Reject` per row. BUYER_MISMATCH only: extra `🔗 Teach alias` |

Type filter chips above the table: `All` / `MISMATCH` / `BUYER_MISMATCH` / `AREA_MISMATCH` / `PRICE_UP` / `PRICE_DOWN`.

Batch actions at the bottom: `[Approve selected (N)]` / `[Reject selected]`. If any selected row had its `New (edit)` cell modified, the user's value is honored.

#### Approve flow (per row)

1. Final value = user-typed override if `New (edit)` was modified, else the proposed DLD value.
2. `UPDATE sales_unit SET <field> = ? WHERE unit_id = ?`
3. `UPDATE pending_change SET disposition='approved', reviewed_at=NOW(), override_value = (user-typed OR NULL) WHERE id = ?`
4. `INSERT INTO audit_log (action = 'override' if user-typed else 'approve', old, new, pending_id, source='review_pending')`
5. Row vanishes from the visible queue; tab count badge decrements live.

#### Reject flow

1. `sales_unit` unchanged.
2. `UPDATE pending_change SET disposition='rejected', reviewed_at=NOW()`
3. `INSERT INTO audit_log (action='reject', old=current, new=NULL, pending_id, source='review_pending')`
4. Row vanishes from queue.

#### Teach-alias flow

Section 2 covers the SQL. UI-wise: click `🔗 Teach alias` → modal confirms "Always treat 'AlGhumlasi' = 'Alghumlasi' for project ONE? (or globally)" → user picks scope → transaction runs → page refreshes.

### Tab 2 — `Drift log`

Read-only — rows where `disposition='auto_applied'`.

| Column | Behavior |
|---|---|
| `When` | `pending_change.reviewed_at` (set at auto_apply time during import) |
| `Unit + Project` | Clickable → opens per-unit History side panel |
| `Field` | E.g., `price` |
| `Old → New` | Side-by-side values |
| `Source` | `DLD import` / `SF import` |

No actions. Default sort: most-recent first. Same project filter as Tab 1.

### Performance

- Pagination at 50 rows per page (both tabs).
- Tab count badges computed once on page load via two indexed `COUNT(*)` queries.
- `New (edit)` input debounced 300ms.
- Approve/Reject buttons disable themselves during the transaction to prevent double-fire.

### Replaces existing flow

- `cmdReviewPending` (in `src/commands/`) no longer builds the HTML+CSV roundtrip. It opens the new page in Electron.
- `cmdApplyPending` is **removed from the sidebar**. The CLI command stays for backward compatibility (with a deprecation notice) for one release cycle; v1.2 may remove it.

## 4. Audit history UX

### Global History page (`📜` sidebar entry)

New sidebar entry between `Tools` and `Backup`. Layout:

```
┌─ History ──────────────────────────────────────────────────────┐
│ Range: [Last 30 days ▼]  Project: [All ▼]  Action: [All ▼]     │
│ Source: [All ▼]   Unit: [_______]   [Apply]   [Export CSV]      │
│                                                                  │
│ ┌────────────────┬────────┬────┬─────────┬─────────┬───────────┐│
│ │ When          │ Project│Unit│ Field   │Old → New│Action·Src ││
│ └────────────────┴────────┴────┴─────────┴─────────┴───────────┘│
│ Page 1 of 14    [‹ Prev]  [Next ›]                              │
└──────────────────────────────────────────────────────────────────┘
```

**Filters:**

- `Range` — `Today` / `Last 7 days` / `Last 30 days` / `Last 90 days` / `All time` / `Custom…`
- `Project` — populated from the same projects-list query used by the top bar
- `Action` — `All` / `approve` / `override` / `reject` / `auto_apply` / `learn_alias`
- `Source` — `All` / `review_pending` / `import_dld` / `import_sf` / `apply_pending`
- `Unit` — free-text exact match on `unit_number`

**Columns** (all sortable): `When` (default desc), `Project`, `Unit` (clickable → side panel), `Field`, `Old → New`, `Action + Source` (stacked).

**Export CSV:** dumps the current filtered set (NOT just the current page) to a `audit-log-YYYYMMDD-HHmm.csv` via save dialog.

**Performance:** Pagination 100 rows/page. Queries backed by `audit_log_project_ts` (for project + range scans) and `audit_log_unit_ts` (for unit filter).

Triggered from any clickable unit cell across the app:

- Review Pending → Tab 1 + Tab 2 (unit column)
- Global History page (unit column)
- Projects page rows
- Status page unit cells

Layout (slides in from the right, 400px wide, overlay):

│   buyer_en:  Ali Alghumlasi        │

│   area:      1,242 sqft            │

**Behavior:**

- Body shows full `audit_log` history for that `unit_id`, newest first, grouped by timestamp.
- `[×]` button + `Esc` key close the panel.
- `[View in global History →]` deep-links to the History page pre-filtered to this unit.
- Renders directly in parent page DOM (no iframe, lightweight flex panel).

**IPC:** one handler `dlp:audit:unit-history` returning `{ current, events }`.

### Item 2 — "All projects" dropdown bug fix

**Symptom:** dropdown shows fewer projects than actually exist.

**Cause:** `src/commands/projects.js` lists only rows from `dld_project`. Projects that have Salesforce data but no DLD snapshot are absent.

**Fix:** Replace the query with a UNION — anything seen on either side qualifies:

```sql
SELECT DISTINCT project_name FROM (
  SELECT project_name FROM dld_project
  UNION
  SELECT DISTINCT sf_sub_project AS project_name FROM sf_sale WHERE sf_sub_project IS NOT NULL
  UNION
  SELECT DISTINCT project_name FROM sales_unit WHERE project_name IS NOT NULL
) ORDER BY project_name;
```

Plus: add a `--source` column in the non-JSON output (`DLD+SF` / `DLD only` / `SF only`) so the asymmetry is visible in CLI mode too.

Also harden the IPC handler in `electron/main.js` — currently `JSON.parse` swallows errors silently and resolves `[]`, meaning a single stderr line silently breaks the dropdown. Fix: route stderr to its own buffer, surface failures with a renderer toast rather than empty-array fallback.

### Item 3 — Import DB summary modal

**Step 1 — Embed metadata in every export.** When `db-export` builds the zip, include a `meta.json`:

```json
{
  "exported_at":  "2026-05-12T14:03:22Z",
  "app_version":  "1.1.0",
  "schema_ver":   "2026-05-12-v2",

```

**Step 2 — On `Import DB`, before applying:**

1. Pick `.zip` (existing flow).
2. Extract to temp directory.
3. Read `meta.json` + open `dld-sync.sqlite` read-only to verify counts.
4. Open a modal:

**Old backups without `meta.json`:** fall back to "Created: unknown, contents not verified" — import still allowed after `[Confirm]`, but the modal shows only current-DB counts (no backup-side counts).

## Out of scope (deferred)

- **Row deletion handling** — when a unit disappears from a fresh import, v1.1 leaves the orphan in `sales_unit`. v1.2 will treat deletes as a new pending_change type.
- **Cross-platform builds** — Windows-only stays. macOS/Linux is a separate release initiative.
- **Audit log retention/archival** — table grows ~12k rows/year, fine for years. Archive policy can wait until size becomes a real concern.
- **Removing the `cmdApplyPending` CLI command** — deprecated in v1.1, removed in v1.2 (gives users one cycle to adapt).
- **Multi-user concurrent edit detection** — DL-Processor is single-user-per-machine; concurrent edit is not a v1.x problem.
