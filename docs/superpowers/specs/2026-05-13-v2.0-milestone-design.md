# DL-Processor v2.0 — Milestone Design

**Status:** scoped 2026-05-12, brainstormed for execution 2026-05-13+
**Branch:** `feat/v2.0-milestone` (off master after v1.2.0)
**Estimated effort:** ~3 weeks subagent-driven execution
**Test target:** 338 → ~390 (+52 new)

## Goal

Ship v2.0 as a unified milestone combining three feature areas that share architecture and reinforce each other:

1. **De-iframe refactor** — replace v1.1's iframe-with-srcdoc pages with native renderer-DOM panes. Foundation for everything else.
2. **Business Process grouping** — replace per-field rows on Review Pending with BP cards grouped by `source_snapshot_id`, classified by SF state (READY / IN_PROGRESS / DLD_ISSUE / REJECTED), filterable by 8 dimensions. Built natively from day 1 on top of (1).
3. **DLD drift detection** — extract `compare.js`'s picker functions into a reusable module so drift detection works for DLD snapshots (not just SF, which was v1.1's scope reduction).

Distributed as the **first patch-based update** to v1.2.0 users via the v1.2 patch system. Proves the patch infrastructure works end-to-end.

## Why bundle these three

- **(1) is the architectural prerequisite for (2).** BP cards work better in native DOM (shared state with sidebar, filter bar, side panel; no iframe boundary). Building BP cards in iframes then de-iframing later = double work.
- **(2) and (3) are operationally coupled.** Both depend on understanding "what changed since last compare" — BP grouping shows the changes, drift detection identifies them.
- **One release vs three.** Cuts ship overhead. Patch system distributes the whole milestone at once. Users get a single coherent v2.0 update.

## Three feature areas

### Feature A — De-iframe refactor

**Files affected:**
- `electron/renderer/tab-host.js` — drop the srcdoc branch. Each tab is a `<div>` element appended to `#tab-host`. Show/hide via `display: block/none`.
- `electron/renderer/review-pending-page.js` — replace `function buildReviewPendingPage(opts) → { html }` with `function renderReviewPendingPage(container, opts)` that builds DOM directly into the container.
- `electron/renderer/history-page.js` — same conversion.
- `electron/renderer/styles.css` — move inline `<style>` blocks from the srcdocs into the main stylesheet with scoped selectors (`.rp-card { ... }` → `.review-pending-page .rp-card { ... }`).
- `electron/renderer/app.js` — update tab-host call sites to pass a container reference instead of an HTML string.

**Behavior change for the user:** none. UI looks identical; just renders without an iframe boundary.

**Behavior change for developers:** scripts run in the main renderer context. `window.parent.dlp.*` becomes just `window.dlp.*`. Click handlers attach to native DOM elements. No more srcdoc HTML string concatenation.

**Risk:** CSS scoping. Existing srcdoc pages use generic class names (`.rp-card`, `.hp-table`). After flattening into one stylesheet, names might collide. Solution: prefix every rule with a page-scope selector (`.review-pending-page`, `.history-page`).

### Feature B — Business Process grouping

Full design carried over from the earlier v1.3 brainstorming. New files all native (no iframes from day 1):

- `src/bp-classifier.js` — `classifyBp(fieldSet) → label`
- `src/sf-state.js` — `classifyState(sfRow) → state`
- `src/commands/review-bps.js` — `listBps`, `approveBp`, `rejectBp`, `acknowledgeBp`
- `electron/renderer/bp-card.js` — card component, renders into a container DOM element
- 4 new IPC handlers (`dlp:review:list-bps`, `:approve-bp`, `:reject-bp`, `:acknowledge-bp`)
- Migration 006 (sf_booking columns) + migration 007 (audit_log.action CHECK)

The Review Pending page (now native after Feature A) renders the BP card list in place of the v1.1 per-row table. Filter bar replaces the simple project dropdown.

### Feature C — DLD drift detection

Refactor `src/compare.js`'s extractor helpers into `src/snapshot-extract.js`:

- `pickLatestPurchase(transactions)` → returns `{ amount_aed, tx_date, ... }`
- `findLatestNonBankParty(transactions)` → returns buyer name
- `pickLatestMarketPrice(transactions)` → returns price
- `extractUnitFields(db, snapshotId, projectId, unitNumberNorm) → { buyer_name, purchase_price_aed, area_sqm, status, procedure_number }` — top-level convenience function called by both compare and drift detection

Update `src/compare-drift.js`:
- The `'dld'` branch (currently a no-op stub from v1.1) becomes real. Loads prev + current snapshot fields via `extractUnitFields`, diffs them, emits `DLD_DRIFT` pending_change rows + `auto_apply` audit_log entries.

Update `src/compare.js`:
- Replace inline picker calls with the new module's functions. No behavior change for MISMATCH detection — just refactored to share code.

Tests: extend `test/compare-drift.test.js` with DLD-side cases mirroring the existing SF-side cases (~6 new).

## Migration plan

`src/migrations/index.js` already runs at every `openDb()`. New migrations 006 + 007 register in the MIGRATIONS array. Idempotent — apply once, skip thereafter.

Existing v1.2 users running compare for the first time after upgrading to v2.0:
- Existing `pending_change` rows back-fill via column defaults (`change_type='MISMATCH'` from migration 003 already)
- Existing `audit_log` entries unaffected
- New compare runs after upgrade write DLD_DRIFT entries when applicable (Feature C kicks in)

## IPC additions (Feature B)

```js
// electron/preload.js — extend the review object
review: {
  list:           (opts)  => ipcRenderer.invoke('dlp:review:list',          opts || {}),
  approve:        (args)  => ipcRenderer.invoke('dlp:review:approve',       args),
  reject:         (args)  => ipcRenderer.invoke('dlp:review:reject',        args),
  teachAlias:     (args)  => ipcRenderer.invoke('dlp:review:teach-alias',   args),
  listBps:        (opts)  => ipcRenderer.invoke('dlp:review:list-bps',      opts || {}),    // NEW
  approveBp:      (args)  => ipcRenderer.invoke('dlp:review:approve-bp',    args),          // NEW
  rejectBp:       (args)  => ipcRenderer.invoke('dlp:review:reject-bp',     args),          // NEW
  acknowledgeBp:  (args)  => ipcRenderer.invoke('dlp:review:acknowledge-bp', args)          // NEW
}
```

`listBps` returns BP objects with `bp_id`, `label`, `state`, `sfContext`, `rows[]` — joins `pending_change` with `dld_project` + latest `sf_booking` row per unit. `approveBp` runs N approvePending calls + writes umbrella audit_log entry, all in one transaction.

## Release

v2.0.0 ships as both:
1. **Full installer** `DL-Processor Setup 2.0.0.exe` for fresh installs or users on v1.1 or older
2. **Patch zip** `dlp-patch-v1.2.0-to-v2.0.0.zip` for existing v1.2.0 users (~5-15 MB)

The patch zip is the first real test of the v1.2 patch system in production. Users who installed v1.2.0 from the team distribution open the app → ⬆ Apply update → pick the zip → restart → v2.0.0.

Cloudflare Pages upload: `.exe` + `latest.yml` for auto-update channel. Manual step on Ali's side.

## Out of scope (defer to v2.1+)

1. **Audit & compliance hardening** — Revert button on global History, user attribution column on audit_log, cryptographic chain, tier-2 approval gates. v2.1.
2. **Workflow automation** — bulk approve, custom rules, anomaly flags. v2.2.
3. **Code signing** — SmartScreen elimination. v2.3.
4. **Direct integrations** — SF API, DLD API. v3.0.

## Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| De-iframe breaks event handlers via CSS scoping leaks | Medium | Per-page scope prefix on every rule; manual smoke after each page conversion |
| BP card filter performance on large pending sets (>500 rows) | Low | Backend `listBps` query is indexed; pagination at 50 cards/page |
| DLD drift detection writes too many drift entries on first run after upgrade | Medium | Idempotency guard already exists (skip if `auto_applied` row already exists for `(snapshot_id, unit, field)`); first compare after upgrade is the seed and won't double-emit |
| Patch zip too large because v2.0 changes app.asar significantly | Low | Asar diff between v1.2 and v2.0 unlikely to exceed 30-40 MB; patch zip compresses to ~10-15 MB |
| First patch distribution surfaces a patch system bug not caught in v1.2 smoke | Medium | Document rollback procedure prominently in TROUBLESHOOTING; Settings → Revert last patch is one click |
