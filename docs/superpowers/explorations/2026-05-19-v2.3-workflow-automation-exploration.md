## Critical blockers for planner

1. **Fix spec path mismatches** in §3.3 of the spec. v2.2 established `electron/renderer/<name>-page.js` at the top level; the v2.3 spec mistakenly assumed a `pages/` subfolder:
   - `pages/automation.js` → `automation-page.js` (new, top-level)
   - `pages/review-pending.js` → `review-pending-page.js`
   - `pages/dashboard.js` → `dashboard-page.js`

2. **Migration 009 — application-layer CHECK validation.** SQLite `CHECK` cannot use `LIKE` or `GLOB` patterns. The spec's requirement to allow `rule:%` and `bulk:%` prefixes in `audit_log.source` must be enforced in code (e.g., a validating helper around `writeAuditLog`) rather than at the schema level. The CHECK constraint can still be widened to an enumerable list, but pattern matching belongs in the app.

3. **Migration 009 — bootstrap audit (cites planning-mistakes-log 2026-05-06).**
   - Audit every read of `pending_change` columns to confirm none assume the old (no-anomaly) shape.
   - Audit every write to `audit_log.source` to confirm the new prefixes are emitted exactly as `rule:<id>` / `bulk:<UUID>`.
   - `writeAuditLog` signature evolution: spec adds `rule_id` indirectly via `source`, not a new column. Confirm in plan.

4. **Confirm SAVEPOINT support in better-sqlite3** before the dry-run feature is planned in detail. The library supports `db.exec('SAVEPOINT name')` + `RELEASE` / `ROLLBACK TO` via raw SQL — confirm with a tiny test before relying on it for the dry-run path.

5. **Migration runner is manual-append** (`src/migrations/index.js` requires explicit array entry). New file `2026-05-18-009-automation.js` won't auto-load; plan must include the append step.

6. **Renderer-DOM rules editor is uncharted UI surface.** v2.2 established the pattern for read-mostly pages (Dashboard, Project Compare). The rules editor is a **write-heavy form** — clause builder, AND/OR chips, save validation. This is the v2.3 equivalent of the v2.1 Settings page complexity; estimate accordingly. (Cites planning-mistakes-log 2026-05-01 — area-branch deferred 3 UI items because the underlying components didn't exist. Same risk if the rules-editor plan assumes generic-form infrastructure that doesn't.)
