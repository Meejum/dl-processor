# DL-Processor v1.2 — Business Process Grouping Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace per-field rows on Review Pending with Business Process cards that group related field changes from the same compare run, label them semantically, and surface SF workflow state inline.

**Architecture:** Reshape Review Pending around BPs. Three new `sf_booking` columns (migration 006). Widened `audit_log.action` CHECK (migration 007). Two pure-function classifiers (BP label, SF state). One new backend command module (`review-bps.js`) exporting `listBps`/`approveBp`/`rejectBp`/`acknowledgeBp` with atomic transactions. New renderer card component (`bp-card.js`) consuming the same srcdoc + inline-script pattern as v1.1's review-pending-page.

**Tech Stack:** Same as v1.1 — Electron 28, better-sqlite3 v11 (Electron-target prebuilt), vanilla HTML/CSS/JS renderer, Node `--test` runner via `scripts/run-tests.js`.

**Spec:** `docs/superpowers/specs/2026-05-12-v1.2-bp-grouping-design.md` (commit `465fa73`).

**Final test target:** 322 → ~360 (+~38 new). Existing 322 stay green throughout.

### Task 7: `approveBp` + `rejectBp` + `acknowledgeBp`

**Files:**
- Modify: `src/commands/review-bps.js` (add 3 new exports)
- Modify: `test/electron/review-bps.test.js` (add 8 cases)

Each is a transactional wrapper over existing `approvePending`/`rejectPending` from v1.1, plus an umbrella `audit_log` write:

```js
function approveBp(db, bpId, overrides = {}) {
  const tx = db.transaction(() => {
    const rows = parseBpId(db, bpId);  // resolves bpId → array of pending_change rows
    for (const row of rows) {
      approvePending(db, row.change_id, overrides[row.change_id] ?? null);
    }
    writeAuditLog(db, {
      projectId: rows[0].project_id, unitNumberNorm: rows[0].unit_number_norm,
      tableName: 'master_data', field: '<bp>',
      oldValue: null, newValue: bpId,
      action: 'approve_bp', source: 'review_pending', changeId: null
    });
  });
  tx();
}
```

Similar for `rejectBp` and `acknowledgeBp`.

Tests: 8 cases — atomic approveBp + per-row overrides; rejectBp marks all rows rejected; acknowledgeBp leaves master_data untouched; partial-failure rollback.

Target: 352 + 8 = 360 passing.
