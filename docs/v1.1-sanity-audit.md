## Findings to fix in v1.1

1. **`history-page.js` project filter dropdown was emitting `value="undefined"` on every option** because `listProjects` did not expose `project_id`. Without `project_id`, the History page Project filter selected `NaN` and matched zero audit rows. **FIXED in this commit:** `src/commands/projects.js` now selects `project_id` from `dld_project` (NULL for SF-only projects); `test/projects-union.test.js` extended with two extra asserts on the new column shape. Existing tests stay green.

2. **`index.js usage()` line 46** still describes `review-pending` as "write pending-changes.csv for review" — that's the pre-v1.1 behaviour. Cosmetic, but visible to anyone who runs `node index.js --help`. **Tiny fix — applied below.**

## Findings to defer to v1.2

1. **Projects table page (`app.js` `buildProjectsPage()`) renders blank columns.** References `p.sf_sub_project`, `p.sf_unit_prefix`, `p.snapshot_count`, `p.last_imported` — none of which exist on the new `listProjects` shape. The new `source` column ("DLD+SF" / "DLD only" / "SF only") is also not surfaced. Fix requires extending `listProjects` with EXISTS-joins to `dld_snapshot` (for `snapshot_count`, `last_imported`) and pulling `sf_sub_project` / `sf_unit_prefix` out of `dld_project`. Deferred — non-blocking, the Status page covers DB-wide counts.
2. **Sidebar "Apply pending" button is still present.** Spec said v1.1 would remove the sidebar entry but keep the CLI. The button + CLI both still work; users following the new inline flow won't hit it. Remove the button + `data-needs-file` wiring in v1.2.
3. **`KNOWN_COMMANDS` in `command-bridge.js` includes `review-pending` and `apply-pending`.** They still resolve through `runSpawn` to the deprecation stub / legacy CSV writer respectively, so nothing is broken. Cleanup candidate when the sidebar button is removed.
4. **History page Source filter is missing `apply_pending` and `compare`** options. Migration 005 widens `audit_log.source` to include `'compare'` (used by SF drift detection); `apply_pending` was in the original enum. The dropdown only exposes `review_pending`, `import_dld`, `import_sf`. Not destructive (audit rows still get written; users just can't filter on those two sources). Add in v1.2 when the filter UI gets a polish pass.
5. **`history-page.js` page-size constant (`PAGE_SIZE = 100`) is inlined in the page-script string.** Hard to tune without rebuilding. Move into a config when the audit log starts hitting real volume.
6. **`unit-history-panel.js` `renderHistory()` does not format `ts` to local time** (it shows the raw SQLite `YYYY-MM-DD HH:MM:SS` UTC string). The two srcdoc pages have a `fmtWhen()` helper for this; the side panel should share it.
7. **No deletion of orphan units from `master_data`.** Spec called this out as deferred to v1.2 ("Row deletion handling"). Stays deferred.
8. **`cmdApplyPending` removal** — same release cycle deferral as the sidebar button.

## Overall

**v1.1 is ready to ship.** The migrations are clean (`b7bb99a` rebuilt schema.sql to v1.1 shape, idempotent on re-runs), the inline Review Pending + Drift Log + History flows are fully wired end-to-end, the per-unit history panel deep-links into the global History page, and the Import DB modal does the safe probe-then-commit dance with a `.bak.<ISO>` snapshot.

Two concrete bugs surfaced during this audit:

- **History page project filter** was completely non-functional — fixed in this commit by extending `listProjects` with `project_id`.
- **`usage()` help text for `review-pending`** is out of date — fixed in this commit.

Two non-blocking issues stay as v1.2 follow-ups: the Projects table page columns and the leftover sidebar `Apply pending` button. Neither blocks the v1.1 release flow.

All 321 tests stay green with the fixes applied.
