Settings round-trip: `loadAppConfig` + `saveAppConfig` operate on `state.appConfigPath = path.join(app.getPath('userData'), 'config.json')` which on Windows resolves to `%APPDATA%\dl-processor\config.json`. `saveAppConfig` uses `mkdirSync(..., { recursive: true })` + atomic-ish write — directory created on first save. `dlp:settings:set` merges into the existing cfg (preserving `dataFolder`, `version`) and writes the whole object back, so the v2.1 keys coexist with v2.0-era keys.

`getTier2Thresholds()` (main.js line 245) reads cfg fresh on each call — no stale-cache concern. Settings changes take effect on the very next approve.

## Backward compat

- **v2.0 → v2.1 migration runs automatically:** migration 008 is appended at the tail of `MIGRATIONS` in `src/migrations/index.js`. `runMigrations` skips migrations whose id is in `schema_migration`, so existing rows from 001–007 are untouched. First launch in v2.1 applies 008 in a transaction.
- **Backfill of existing audit_log rows:** Step C walks rows where `row_hash IS NULL` in `(ts, audit_id)` order. Empty audit_log → no-op. Partially-chained audit_log → continues from the last `row_hash IS NOT NULL` row rather than restarting from genesis.
- **NULL `user` on old rows stays NULL:** Step A adds `user TEXT` (no default) and `tier2 INTEGER NOT NULL DEFAULT 0`. Existing rows get `user = NULL` (forward-only attribution, per spec) and `tier2 = 0`.
- **First v2.1 action writes new fields:** `writeAuditLog` resolves `user` via `currentUser()` when caller passes no explicit value. `currentUser` cascades: Settings `audit_user` → `os.userInfo().username` → `'unknown'`.
- **Settings modal tolerates missing keys:** `dlp:settings:get` applies defaults via `cfg.X != null ? cfg.X : DEFAULT` — pre-v2.1 config.json files (with only `dataFolder` + `version`) yield defaults of 10 / 50000 / 5. First save adds the new keys without disturbing the existing ones.
- **REVERTABLE_ACTIONS gate:** Revert button is hidden for non-master_data rows (e.g. learn_alias targeting buyer_alias) and for actions outside the set — old auto_apply rows in audit_log get no Revert affordance, by design.

## Findings to fix in v2.1

1. **history-page.js Action filter was missing 4 valid actions** — added `approve_bp`, `reject_bp`, `acknowledge_bp`, `revert` to the `<select class="hp-action">` dropdown. Without these, users could only filter audit_log to the v1.1 actions; BP-level decisions and reverts were invisible to the filter even though the rows showed up under "All". **Fixed inline (this commit).**

## Findings to defer to v2.2

1. **`src/current-user.js` `appConfigPath` export is unused externally.** main.js resolves the same path through `app.getPath('userData')` instead. Either prune the export, or refactor main.js to use the shared helper for consistency. Harmless either way.
2. **Tier-2 logic duplicated between `src/tier2.js` and `electron/renderer/review-pending-page.js`.** Renderer copy is necessary today (no `require()` from src/ inside the renderer). A v2.2 step could expose `isTier2` via a preload IPC (e.g. `dlp:tier2:check`) so the renderer doesn't have to maintain a parallel implementation. Backend already re-checks defensively, so a desync would only mis-trigger the modal — never silently bypass tier-2.
3. **REVERTABLE_ACTIONS duplicated between `src/audit-fields.js` and `electron/renderer/history-page.js`.** Same root cause as above. Same v2.2 follow-up shape — expose via a preload constant or IPC.
4. **`audit_log.user` is `TEXT` not `TEXT NOT NULL`.** Forward-only attribution is correct (old rows stay NULL), but new rows always have a user resolved (Settings → OS → 'unknown'). v2.2 could add a `CHECK` constraint that only NEW rows after migration 008 must have user IS NOT NULL — not trivial in SQLite, deferrable.
5. **Settings modal has no validation on tier-2 number inputs.** Empty / NaN values get coerced via `Number('')` = 0 by the Save handler. A 0 threshold effectively disables that tier-2 dimension. Add min/required validation or treat `null` as "use default" instead of coercing to 0.

## Overall verdict

v2.1 is ready to ship. The cryptographic chain primitives, migration 008, revert workflow, tier-2 justification flow, Settings round-trip, and Excel export are all wired end-to-end with matching backend handlers, preload bridges, and renderer event handlers. The 422-test suite passes both before and after the one inline fix made during this audit. Backward compatibility with v2.0 databases is preserved by appendind 008 at the migration tail and treating NULL `user` columns on legacy rows as the documented forward-only-attribution model. CLI compatibility is preserved. The remaining items above are minor polish items appropriate to defer to v2.2 — none gate the v2.1 release.
